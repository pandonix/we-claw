# 风险与漂移跟踪

> 用途：在 `02-实施TODO.md` 完成首期清单后，集中跟踪启动开发/测试前后的剩余风险、文档与实现的不一致，以及容易在后续迭代中发生开发漂移的设计点。
>
> 状态约定：`[ ]` 待办、`[~]` 进行中、`[x]` 已解决、`[!]` 已确认但暂不修复（需在条目里写明降级方案）。
>
> 修改条目时请同步更新 `last-touched`。

## 1. 立即风险（开发/测试启动前应先收敛）

### R1. `permissionMode` 取值与 SDK 不一致
- **状态**：`[ ]`
- **严重度**：高
- **证据**：
  - `src/launcher/config.ts:11,42` 把 `dontAsk` / `auto` 列为合法值，默认值仍是 `dontAsk`。
  - `src/launcher/runtime-bridge.ts:386` 直接把它透传到 SDK `Options.permissionMode`。
  - `@anthropic-ai/claude-agent-sdk` 实际只接受 `default | acceptEdits | bypassPermissions | plan`。
  - `01-集成基线与实施建议.md` §4 风险条也写"默认使用 `dontAsk`"，与官方 API 直接冲突。
- **影响**：首条 prompt 行为与设计预期不符，可能被 SDK 静默丢弃或抛错。
- **建议处置**：在 adapter 内把 `dontAsk` / `auto` 显式映射成 SDK 支持的枚举（例如 `dontAsk → bypassPermissions`），并同步修订 `01` 文档。
- **last-touched**：2026-04-30

### R2. SDK 仍在模块顶层 import，optional 依赖语义不闭环
- **状态**：`[~]`
- **严重度**：中
- **证据**：
  - `package.json` 已将 `@anthropic-ai/claude-agent-sdk` 移到 `optionalDependencies`。
  - 但 `src/launcher/runtime-bridge.ts:8` 仍是顶层静态 `import`，模块加载时如果 optional dep 缺失，整个 launcher 会崩溃，OpenClaw-only 部署不可用。
- **影响**：optional 化只在打包层面成立；运行时仍是硬依赖。
- **建议处置**：把 SDK 调用收敛到一个延迟加载的模块（`await import(...)` 或在选定 `claude-agent-sdk` runtime 时再 require），并在缺失时降级为 bootstrap diagnostic。
- **last-touched**：2026-04-30

### R3. `sessions.subscribe` / `sessions.messages.subscribe` 是哑实现
- **状态**：`[ ]`
- **严重度**：中
- **证据**：
  - `src/launcher/runtime-bridge.ts:234` 直接返回 `{ subscribed: true }`，没有真正注册推送。
  - `src/launcher/runtime-bridge.ts:367` 仅在 `consumeRun` 结束时 emit 一次 `sessions.changed`。
  - 前端 `src/app/main.ts:132,784` 已经在订阅，OpenClaw 路径会真正推送，Claude 路径不会。
- **影响**：Claude runtime 下 UI 中途不刷新会话列表/活动状态，需手动 reload 才能看到变化。
- **建议处置**：要么补一个本地 emitter，把 `localSessions` 变更主动广播；要么在能力模型里把订阅类方法标记为 unsupported，前端按 capability 降级（不再下发订阅）。
- **last-touched**：2026-04-30

### R4. tool 事件只发 start，没有 update / completed / error
- **状态**：`[ ]`
- **严重度**：中
- **证据**：
  - `src/launcher/runtime-bridge.ts:439-462` 只 emit `phase: "start"`。
  - 前端 `ToolBlockStatus` 支持 `running | updated | completed | error`，长任务会出现工具行永远停在 running。
- **影响**：UI 视觉与实际状态不符，验收清单未列入此项。
- **建议处置**：从 SDK `tool_result` block 派生 completed / error；同时在 `consumeRun` finally 里兜底关闭悬挂的 tool block。
- **last-touched**：2026-04-30

### R5. history reload 丢非纯文本消息
- **状态**：`[ ]`
- **严重度**：中
- **证据**：
  - `src/launcher/runtime-bridge.ts:127-144` 过滤 `text.trim()` 为空的 entry，tool_use-only / thinking / subagent 边界消息全部消失。
- **影响**：流式期间能看到工具行，刷新历史后看不到——典型的"看起来像 bug"的不一致。
- **建议处置**：把 tool_use / subagent 消息映射成 compact tool row（与 R4 共用一份归一化逻辑），而不是直接过滤。
- **last-touched**：2026-04-30

### R6. abort 后的 UI 状态收敛依赖 `error.name === "AbortError"`
- **状态**：`[ ]`
- **严重度**：中
- **证据**：
  - `src/launcher/runtime-bridge.ts:362` 用名字字符串判定 abort。
  - `docs/hermes-integration-impact-analysis.md:136,509` 也提示"取消语义取决于 SDK 当前版本"。
- **影响**：SDK 升级若改用 `DOMException` 或别名，UI 永远停在 running。
- **建议处置**：同时识别 `AbortError`、`controller.signal.aborted === true`、SDK 文档列出的 abort 派生类；在 `consumeRun` finally 里强制 emit 一个最终态。
- **last-touched**：2026-04-30

### R7. 没有 API key / 认证预检
- **状态**：`[ ]`
- **严重度**：中
- **证据**：
  - `src/launcher/runtime-bridge.ts:84-89` 仅靠 SDK 包是否安装判断 `available`。
  - 验收清单 §4 备注承认需要本机 Anthropic API key，但 bootstrap 没有相应 diagnostic。
- **影响**：UI 显示"已连接"，但发第一条 prompt 即报 401/缺 key，归因体验差。
- **建议处置**：bootstrap 阶段做一个轻量自检（只检测 `ANTHROPIC_API_KEY` / 受支持云供应商凭证是否非空，不读密钥本身），缺失时下发 `runtime.auth-missing` diagnostic。
- **last-touched**：2026-04-30

### R8. OpenClaw 回归未跑过
- **状态**：`[ ]`
- **严重度**：高
- **证据**：
  - `03-验收清单.md` §2 三项全是 `[ ]`：默认 runtime 未设置仍连 OpenClaw、`/api/gateway/ws` 兼容、`/api/runtime/ws` 在 OpenClaw 下复用 Gateway bridge。
  - 当前 runtime bridge 改造把 `/api/runtime/ws` 接管所有 upgrade（`runtime-bridge.ts:39-50`），如果路径分发出错，OpenClaw 路径会静默坏掉。
- **影响**：现有 v1 交付行为可能已经回退但未被发现。
- **建议处置**：在补 Claude 手动验收前，先把 §2 三条跑通并写一条最小 e2e 测试。
- **last-touched**：2026-04-30

## 2. 设计灰色地带（容易在后续迭代中漂移）

### D1. pending → persisted session key 升级语义未定义
- **状态**：`[ ]`
- **证据**：
  - `src/launcher/runtime-bridge.ts:371-376` 抓到真实 `session_id` 后只更新 `localSession.sessionId`，**没有改 sessionKey**，也没把 `localSessions` 的 Map 键迁移。
  - 文档定义了 `claude:pending:<uuid>` 与 `claude:<sessionId>` 两类前缀，但没有说明何时/如何升级。
- **风险**：刷新页面后 UI 用 `claude:<sessionId>` key，launcher 内存仍是 pending key，导致同会话重复出现/状态错乱。
- **建议处置**：在抓到 sessionId 时同步把 Map 键改为 `claude:<sessionId>`，并把升级规则写进 `01` 文档。
- **last-touched**：2026-04-30

### D2. resume 与 cwd 强绑定，但 UI 不表达 cwd
- **状态**：`[ ]`
- **证据**：
  - `runtime-bridge.ts:258,290,383` 一律用 `this.context.config.claudeSdkCwd`。
  - `docs/hermes-integration-impact-analysis.md:418-426` 已经指出 session 应该按 `runtimeKind + cwd + sessionId` 分组。
- **风险**：用户跨 cwd 切换时会以为"会话丢了"；多项目下 session rail 没有可视分组。
- **建议处置**：先在 statusbar/session subtitle 显式展示当前 cwd；后续做多 cwd 支持时再扩 rail 分组。
- **last-touched**：2026-04-30

### D3. `WE_CLAW_RUNTIME=auto` 实际就是 openclaw alias
- **状态**：`[!]`
- **证据**：`src/launcher/config.ts:38`、`src/launcher/runtime-bridge.ts:53`。
- **风险**：配置项看起来"自动选择"，实际是固定 alias，未来加策略时会被误以为已存在。
- **降级方案**：暂不改行为，但需要在 `01` 文档与 README 显式注明"当前 `auto` 等价于 `openclaw`"。
- **last-touched**：2026-04-30

### D4. capabilities 是写死常量
- **状态**：`[ ]`
- **证据**：
  - `src/launcher/bootstrap.ts:91-102` OpenClaw capabilities 全部写死，未从真实 OpenClaw bootstrap 推导。
  - `runtime-bridge.ts:501-529` Claude capabilities 也是常量。
- **风险**：未来接 Hermes / SDK V2 时，"加一个 runtime = 改一处常量"模式很容易演化成 UI 内部 if-else 的源头。
- **建议处置**：把 capabilities 提取成 adapter 内部 `describe()` 方法，bootstrap 读取真实 adapter 的能力，便于后续按 capability 在 UI 控制 create/abort/approval 等控件。
- **last-touched**：2026-04-30

### D5. launcher 重启 pending session 丢失（已知）
- **状态**：`[!]`
- **证据**：`01-集成基线与实施建议.md` §4 第三条；TODO §5 未排期。
- **降级方案**：首期接受。需要在 README / `03` 验收清单 §4 显式注明，避免被当成新 bug 反复上报。
- **last-touched**：2026-04-30

## 3. 测试覆盖缺口

### T1. 自动化测试只覆盖了 normalizer
- **状态**：`[ ]`
- **证据**：`test/runtime-bridge.test.ts` 仅 3 个 case，覆盖 `normalizeClaudeSession` / `normalizeClaudeHistoryMessages` / `claudeStreamDeltaText`。
- **缺口**：`consumeRun` 状态机、abort 路径、`captureSessionId` + Map 升级、tool 事件、resume 行为均无测试。

## 4. 优先级建议（开发/测试启动前的最小收敛序）

1. R1（permissionMode 合法值）→ 否则首条 prompt 行为不可预期。
2. R8（OpenClaw 回归）→ 证明 runtime bridge 改造没破坏 v1 交付。
3. R2（顶层 import 延迟化）→ 把 optionalDependencies 的语义补完整。
4. R3（订阅哑实现降级或补推送）→ 让 UI 行为与 capability 一致。
5. R7（auth 预检 diagnostic）→ 让"连得上但发不出"变成显式提示。
6. R4 + R5（tool 生命周期一致性）→ 避免 UI 出现"流式 vs 历史"不一致。
7. T1 / T2 → 在以上六项收敛后再补回归测试与手动验收。

完成 1–5 之前不建议进入集成测试阶段。
